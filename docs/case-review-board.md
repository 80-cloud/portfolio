# review-board — 相互レビューコミュニティ（ケーススタディ）

学習者どうしが成果物を多軸で評価し、コメントと合格バッジで成長を支え合う Web アプリ。
重点は **「誰が・何に・何をできるか」＝認可と安全性** の設計です。

- コード: https://github.com/80-cloud/review-board
- 設計・運用ドキュメント: https://github.com/80-cloud/review-board-docs （脅威モデリング／インフラ／テスト計画／品質など）
- ライブ: 正規TLS（Let's Encrypt・自動更新）で本番稼働中。ログインは招待制のため、登録不要で触れるデモ環境は別途準備中（選考時に画面共有も可能）。
- 担当: 個人開発（要件・設計・実装・AWS 運用まで） ／ 期間: 2026-05〜

---

## 解きたい課題

IT 未経験の学習者にとって、書いた成果物へ「安心して見せ合い、建設的なフィードバックを受けられる場」は意外と少ない。
公開の場は心理的な壁が高く、逆に個別のやり取りは記録が残らず成長が見えにくい。

そこで **「同じクラス（期）の中だけ」という閉じた範囲**で、投稿→相互レビュー→合格バッジまでを記録として残せる場を作りました。
ただし「閉じた範囲」を名乗る以上、**その範囲が本当に守られていること**——他人・他クラスの情報が漏れない・書き換えられないこと——が成立の条件になります。
このアプリの中心はそこにあります。

---

## アプローチと差別化

| 差別化 | 中身 |
|---|---|
| 認可を中心に据えた設計 | 「ログイン → 同じクラス → 本人／役割」の 3 段を **必ずサーバー側で**確認。画面の出し分けに頼らない |
| 「できてはいけない」を明示 | 権限一覧（マトリクス）を先に定義し、それを**テストで裏取り**。仕様と検証が 1:1 |
| 存在を悟らせない応答 | 見えてはいけないものは 403 でなく **404**（「禁止」と返すと存在がバレる）。操作不可は 403、未認証は 401 と使い分け |
| 多層防御（ひとつの対策に頼らない） | 通信暗号化・安全な Cookie・最小権限・機密の外部化・削除保護を層で重ねる |
| 正直に記録する | できている／改善中／現時点は対象外を隠さず記録。未対応を「やらない」でなく「今のフェーズでは簡素化」と明示 |

## 重点：認可と安全性の設計（このアプリの核）

### 3 段の信頼境界

リクエストが来たら、次の順にサーバーで確認し、どれかを満たさなければ通しません。

1. **ログインしているか（認証）**
2. **同じクラス（期）の人か（範囲）**
3. **本人か／どの役割か（権限）**

### 権限マトリクス（抜粋）

| 操作 | 未ログイン | 受講生 | 講師 | 管理者 |
|---|:--:|:--:|:--:|:--:|
| 同じクラスの投稿を見る | × | ○ | ○ | ○ |
| 自分の投稿を編集・削除 | × | ○ | ○ | ○ |
| 他人の投稿を編集・削除 | × | × | × | × |
| 成果物の最終評価（合格・差し戻し） | × | × | ○ | × |
| 他クラスの情報を見る | × | × | × | × |
| クラス作成・アカウント発行 | × | × | × | ○ |

役割ごとに「何ができて何ができないか」を**先に表で定義**し、権限の判断ロジックは 1 か所に集約しています。

### なぜ 404 を返すのか

見えてはいけないリソースに 403（禁止）を返すと「存在はする」と分かってしまう。
404（ありません）なら**存在自体を悟らせない**。この「見えない＝404／できない＝403」の一貫方針で、他人のデータを ID 直指定でのぞく種類の攻撃（IDOR）を設計段階で塞いでいます。

### 認証・セッション

- パスワードは元に戻せない形（ハッシュ化）で保存
- ログイントークンは **短命アクセス＋使い捨てリフレッシュ＋再利用の検知**
- ワンタイムパスワード（TOTP）による二要素認証に対応
- Cookie は外部サイトから使われない・盗まれにくい安全な設定

### 監査ログの改ざん検知

「だれが・いつ・何にアクセスしたか」を記録し、**後から書き換えられていないかを検証できる**形（ハッシュでつなぐ）で残しています。

### アップロード画像の安全化

拡張子でなく **中身（先頭の数バイト）で種類を判定**し、非公開で保管。閲覧時だけ**短時間で切れる URL**を発行します。

## 品質をどう担保しているか

### 「できる」だけでなく「できてはいけない」をテストする

このアプリの肝は「他人・他クラスにアクセスできないこと」。そこで**拒否されること**を必ずテストします。

- 他人の投稿は編集・削除できない（できたら不合格）
- 別クラスの投稿・成長記録は見えない
- 受講生は最終評価（合格／差し戻し）ができない（講師だけ）
- 未ログインでは保護 API を使えない

### CI で毎回自動検査（変更のたびに実行、通らないものは取り込まない）

| 種類 | 内容 |
|---|---|
| 単体／結合／E2E | 部品単体・DB 込みの結合・画面からの通し操作 |
| 性能テスト | 「この速さを下回ったら不合格」というしきい値で合否判定 |
| 依存脆弱性スキャン | 使っている部品に既知の弱点が無いか |
| 機密情報スキャン | パスワードや鍵をコードに含めていないか |
| 静的解析（SAST） | 危険なコードパターンの検出 |
| 部品一覧（SBOM） | 何を使っているかを記録し、後から弱点を追える |

- アクセシビリティにも配慮（ラベル・コントラスト・読み上げ）し、主要画面で高スコアを確認。
- カバレッジは学習段階では合否ゲートにせず、参考値として不足箇所の手がかりに使用。

## インフラと運用

```
[ブラウザ] ──HTTPS──▶ [受付(nginx)] ──▶ [アプリ(Spring Boot)] ──▶ [DB(PostgreSQL・非公開)]
                                                              └──▶ [画像ストレージ(S3互換・非公開/一時URL)]
```

- 構成は **Terraform でコード化（IaC）**。手作業を減らし同じ環境を再現できる。
- DB は外から直接触れない場所（プライベート）に置き、アプリ経由でのみアクセス。
- アプリには最小権限のみ付与。機密はコードに書かず安全な保管場所から渡す。
- 予算アラートで想定外の課金に早く気づく。
- **誤操作防止**：重要な資源は簡単に消せない設定（削除保護）＋取り返しのつかない操作は人の確認を必ず挟む。

## 本番運用で踏んだ問題と対処（実際に直したこと）

作って終わりでなく、本番で動かして踏んだ問題を自分で切り分けて直しました。

| 起きたこと | 原因 | 対処・学び |
|---|---|---|
| 再デプロイ後に自動デプロイが署名エラーで停止 | 作り直しでクラウドの資格情報（アクセスキー）が新規発行され、古い値のまま | 新しい値へ更新して復旧。以後「作り直すと資格情報が変わる」前提で鍵の更新を手順に組み込み |
| 自動デプロイのワークフローが起動しない | 条件式から機密（secrets）を直接参照できない仕様を見落とし | 条件の書き方を修正。CI の実行モデルを理解して回避 |
| 既存 DB で構造変更が失敗 | マイグレーションの整合（チェックサム）ずれ | 差分を切り分けて解消。スキーマ変更は管理ツール経由に統一 |
| ブラウザに証明書警告 | 生 IP の自己署名 TLS で稼働開始 | **無料のドメイン（DuckDNS）＋ Let's Encrypt で正規 TLS ＋自動更新へ移行完了。追加費用ゼロで警告を解消** |

> 「本番は一度で完成しない」を体験し、**再現性（IaC）・自動検査（CI）・記録（監査ログ）**がそのまま復旧力になると実感しました。

## 拡張に強い構造

- **役割で分けたレイヤ**（受付 / 業務処理 / データ）。権限の判断は 1 か所に集約。
- **エラー処理を一元化**（どこで失敗しても同じ作法で返す）。
- データは消さず **論理削除**で残し、履歴と整合を保つ。
- 画像は外部ストレージに置き、DB を重くしない。

---

## 設計判断の記録（なぜそう作ったか）

あとから検証できるよう、主要な判断の理由を残しています。

- 見えない＝404／できない＝403 に統一した理由（存在秘匿と一貫性）
- 認可を画面でなくサーバーの 1 か所に集約した理由（出し分け依存の穴を無くす）
- 冗長化を今は入れず 1 台構成にした理由（後述のトレードオフ）
- 画像を DB でなく外部ストレージ＋一時 URL にした理由（DB 負荷と漏えい範囲の最小化）

---

## 現時点で対象にしていないこと（トレードオフを正直に）

「やらない」ではなく、**今のフェーズで意図的に簡素化**している点です。本格運用に移る際に見直します。

- **サーバー 1 台構成**（冗長化・自動切替は今後の課題）。1 台に問題が起きると一時的に止まりうる。→ 無料枠中心のコスト方針とのトレードオフ。
- **カバレッジの合否ゲート**は未導入（数値は参考値として把握）。
- 外部公開向けの追加防御層（前段の WAF 等）は最小限。

---

## ロードマップ

- **冗長化（複数 AZ）** による可用性向上（現在は 1 台構成）。
- 登録不要の**デモ環境**の公開（架空シードのクラスで全画面を触れるようにする）。
- メール通知のダイジェスト化・配信停止など、通知まわりの拡充。

> 正規ドメイン＋自動更新 TLS（DuckDNS ＋ Let's Encrypt）は移行完了済み。
